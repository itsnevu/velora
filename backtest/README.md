# backtest — offline strategy validator

A tiny, **dependency-free ES-module** backtester for the Robinhood Agentic desk. It lets
the PM sanity-check a strategy's *rules* against historical daily bars **before** ever
proposing a live trade — entirely offline, on plain `node`, with **no npm install**.

> ⚠️ **Illustrative only. NOT a track record. NOT financial advice.**
> The committed fixtures are *synthetic* price series generated by a deterministic
> formula specifically to exercise each strategy's code paths. Any numbers you see here
> (returns, win rate, Sharpe, …) describe made-up data. Feeding it real bars produces a
> *hypothetical, in-sample* result with look-ahead-free fills — it still is **not** a
> promise of live performance, and it never places an order or touches an account.

## Why it exists

`CLAUDE.md` requires every proposed trade to map to a written rule in `strategies/`.
This module encodes two of those rule files as executable signal functions and replays
them bar-by-bar so the desk can see, concretely, *when* a strategy would enter/exit and
*how* the risk caps in `strategies/README.md` bind. It is a rule checker, not an
optimizer or a return generator.

## Layout

```
backtest/
  indicators.js                 pure OHLCV indicators (sma/ema/rsi/atr/rolling*/stddev/bollinger)
  engine.js                     event loop: next-open fills, risk sizing, caps, stops -> report
  metrics.js                    report metrics + the compact dashboard summary
  strategies/
    mean-reversion.js           encodes strategies/mean-reversion.md
    left-side-accumulation.js   encodes strategies/left-side-accumulation.md
  fixtures/
    uptrend.json                synthetic rising-50DMA series with dips (mean-reversion)
    drawdown.json               synthetic -30% fear selloff into support (left-side)
  run.js                        CLI: run a strategy over a --bars file, write a report
  test.js                       no-dep assert runner (indicators + fixture invariants)
  reports/                      generated reports (gitignored; may hold real-symbol runs)
```

## Run it

```bash
node backtest/test.js            # asserts indicators + fixture invariants; exits non-zero on failure

node backtest/run.js --strategy mean-reversion --bars backtest/fixtures/uptrend.json --desk
node backtest/run.js --strategy left-side-accumulation --bars backtest/fixtures/drawdown.json
```

`run.js` writes the full report to `backtest/reports/<strategy>.json` and prints a terse
summary table. `--desk` additionally emits the **compact** summary the dashboard embeds in
`desk-state.json` (printed, and written to `backtest/reports/<strategy>.compact.json`).

### CLI options

| flag | meaning |
|---|---|
| `--strategy <name>` | `mean-reversion` \| `left-side-accumulation` (aliases `mr`, `left-side`) |
| `--bars <path>` | JSON array of daily OHLCV bars (see below) |
| `--symbol <sym>` | label stamped into the report (default derived from the strategy) |
| `--out <path>` | report destination (default `backtest/reports/<strategy>.json`) |
| `--desk` | also emit the compact `desk-state.json` summary shape |
| `--risk <pct>` | override risk-per-trade % (defaults: 1 mean-rev, 2 left-side) |
| `--start <usd>` `--commission <usd>` `--slippage <bps>` | engine knobs (defaults 10000 / 0 / 5) |

## Feeding real bars (live use)

The engine is data-agnostic. In production the PM pulls real daily bars for a symbol via
the `robinhood-trading` MCP tool **`get_equity_historicals`**, saves them to a JSON file
shaped exactly like the fixtures, and points `--bars` at it:

```jsonc
// bars/AAPL.json  (oldest -> newest, one object per trading day)
[
  { "date": "2025-01-02", "open": 185.1, "high": 187.4, "low": 184.6, "close": 186.9, "volume": 52131000 },
  ...
]
```

```bash
node backtest/run.js --strategy mean-reversion --bars bars/AAPL.json --symbol AAPL --desk
```

The backtester **only reads that file** — it does not call MCP, does not read positions,
and cannot place orders. It is a pure offline computation.

```jsonc
{
  "schemaVersion": 1,
  "strategy": "mean-reversion" | "left-side-accumulation",
  "symbol": "DEMO-UPTREND",
  "generatedAt": "2026-07-11T00:00:00Z",   // engine default is a FIXED date; run.js stamps real time
  "params": { "startingEquityUsd": 10000, "riskPerTradePct": 1, "perTradeCapPct": 15,
              "concentrationCapPct": 25, "stopLossPct": 8, "commissionUsd": 0, "slippageBps": 5 },
  "period": { "from": "YYYY-MM-DD", "to": "YYYY-MM-DD", "bars": 230 },
  "metrics": { "totalReturnPct", "buyHoldReturnPct", "trades", "winRatePct", "avgWinPct",
               "avgLossPct", "profitFactor", "maxDrawdownPct" /* <=0 */, "avgHoldDays",
               "exposurePct", "sharpe" },
  "equityCurve": [ { "date": "YYYY-MM-DD", "equity": 10000 }, ... ],  // one per bar

}
```

### Compact summary (`desk-state.json` → `backtests[]`)

`--desk` emits the shape the dashboard reads:

```jsonc
{ "strategy", "symbol", "period": { "from", "to", "bars" },
  "metrics": { ...same keys as above... },
  "equitySpark":  [ ~40 downsampled equity values ],
  "buyHoldSpark": [ ~40 downsampled buy&hold-equity values ] }
```

The PM merges each compact object into `desk-state.json` under an array named
`"backtests"` after a desk run.

## What the strategies encode

**`mean-reversion.js`** (see `strategies/mean-reversion.md`) — buy oversold pullbacks
inside an uptrend. Entry when: price above a **rising 50DMA**, a pullback tags a defined
support (the 50DMA or the lower Bollinger band — the .md's "RSI≤30 **or** tagging
support" OR-clause), and the bar **stabilizes** (not a fresh lower-low crash). Exit on RSI
≥ 55 / 20DMA reclaim (`target`), a close back below the 50DMA (`trend-break`), the 10-bar
`time` stop, or the −8% hard `stop`. One entry per setup, one position at a time — no
averaging.

**`left-side-accumulation.js`** (see `strategies/left-side-accumulation.md`) — the
*defined exception* to "no averaging into losers": a **pre-planned two-tranche ladder**
(60% / 40%) at named support inside a ≥20% drawdown with RSI≤30 (or a −2σ band tag), a
whole-position kill-stop, total position-risk ≤ 2% of NAV, concentration ≤ 25%. The full
plan (both levels, both sizes, the kill-stop) is fixed **before** tranche 1; risk never
grows.

## Known simplifications & caveats (read before trusting a number)

- **Synthetic fixtures.** `uptrend.json` / `drawdown.json` are deterministic
  index-driven series built to *trigger* the rules, not realistic tape. Their clean,
  low-noise shape flatters risk-adjusted stats (e.g. Sharpe) — do not read anything into
  them.
- **Daily bars, no intraday.** One decision per bar on the **close**.
- **Next-bar-open fills.** A signal on bar *i*'s close fills at bar *i+1*'s open — this is
  how the engine avoids look-ahead. The last bar cannot generate a fill; any position
  still open at the end is force-closed at the final close (`exitReason: "end"`).
- **Stops evaluate on the close.** A bar that *closes* at/below the stop triggers a
  next-open exit; intraday wick touches are ignored (the left-side .md likewise prefers a
  weekly close). No intrabar stop fills.
- **Fundamental / macro gates are NOT modeled.** Both .md files require checks a price
  series can't express — fundamental score, "fear vs thesis-break", earnings blackout,
  risk-off backdrop, injection alerts. Those stay the live desk sub-agents' job; the
  backtester models only the **price** structure. A green backtest is necessary, not
  sufficient.
- **Single concurrent position.** Mean-reversion is one entry per setup. Left-side allows
  exactly one planned **add** (T2) to the same position; scale-**out** is collapsed to a
  single full exit on the first strength signal (rather than trimming in pieces), which is
  conservative for reporting one P&L per trade.
- **Whole-share sizing.** Shares are floored to integers; the fractional-share nuance
  Robinhood supports is not modeled. Left-side substitutes its wider, 2%-risk-budgeted
  kill-stop for the generic −8% per-position stop, exactly as that strategy prescribes.
- **`profitFactor` sentinel.** With zero losing trades profit factor is reported as
  `99.99` (JSON has no `Infinity`).
- **Not an optimizer.** No parameter search, no walk-forward, no transaction-cost
  calibration beyond the flat commission + bps slippage you pass in.
